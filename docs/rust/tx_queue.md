# Диспетчеризация и StealthEngine (TxQueue)

Модуль `tx_dispatch` отвечает за отправку сгенерированных 802.11 кадров в сеть. Архитектура отправки спроектирована с учетом предотвращения Head-of-Line Blocking и минимизации вероятности обнаружения системами WIDS/WIPS (Wireless Intrusion Detection Systems).

## 1. Типы Фреймов (TxFrame)

Все исходящие пакеты типизируются через `enum TxFrame`, что позволяет ядру маршрутизировать их с разным приоритетом:

```rust
// core/src/engine/tx_dispatch.rs
pub enum TxFrame {
    Normal(Vec<u8>),       // Фоновый трафик (Beacon Flood)
    Priority(Vec<u8>),     // Критический трафик (Targeted Deauth, Response)
    Batch(Vec<Vec<u8>>),   // Амортизированная отправка (recvmmsg/sendmmsg batch)
}
```

- Разделение очередей на **Normal** и **Priority** критично для атак типа Evil Twin. Ответ (Probe Response) на запрос клиента должен быть отправлен в `Priority` очередь, чтобы опередить легитимную точку доступа.

## 2. Алгоритм Fair Drain

Вместо простой FIFO-очереди, `TxQueue` реализует модифицированный Round-Robin с весами (Weighted Fair Queuing). Если активно выполняется ресурсоемкая атака (например, рассылка Deauth), `TxQueue` гарантирует, что 20% эфирного времени (по умолчанию) всегда останется под ответы (Probe Responses) из приоритетной очереди.

Эта логика выполняется в выделенном потоке-отправителе, который читает данные из MPSC-канала:
```rust
// Заглушка (в процессе реализации Phase 4)
pub struct TxQueue;
```
В Phase 4 модуль будет использовать `crossbeam::channel::select!` для балансировки нагрузки между Priority/Normal каналами.

## 3. StealthEngine (Маскировка)

`StealthEngine` — это промежуточный (middleware) слой между `TxQueue` и `af_packet` отправкой (socket `send()`). Он выполняет модификацию пакетов *на лету*.

:::danger
**Strict Compliance Statement**: Использование функций `StealthEngine` предназначено исключительно для стресс-тестирования *собственных* систем мониторинга инцидентов (SIEM) с целью проверки корректности их работы (генерации ложноположительных и ложноотрицательных срабатываний).
:::

Функции модуляции трафика:
1. **OUI Spoofing (MAC-маскировка)**: MAC-адрес отправителя (TA) циклически ротируется. Изменение затрагивает только 3 последних октета (NIC часть), сохраняя валидный OUI производителя (например, Apple или Samsung) для обхода фильтров.
2. **Jitter Injection**: Строго периодические кадры (например, отправка Deauth ровно каждые 100 мс) мгновенно помечаются WIDS-системами как атака. `StealthEngine` добавляет математический джиттер (например, $\pm 15$ мс) к интервалам засыпания (`thread::sleep`).
3. **Burst Capping**: Ограничение максимального количества кадров в секунду (Token Bucket механизм) для предотвращения физического отказа в обслуживании (Denial of Service), удерживая нагрузку в рамках аудита, а не DoS-атаки.
