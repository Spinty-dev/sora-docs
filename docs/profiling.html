Методология Сквозного Профилирования (Profiling)

Для обеспечения производительности на уровне 10Gbps и работы с высоконагруженными Wi-Fi сетями требуется глубокий анализ задержек на каждом этапе обработки.

1. Профилирование Rust-ядра (CPU & Memory)​

Для поиска узких мест в парсинге пакетов (parsers.rs) и управлении очередями (packet_engine.rs) используется инструмент flamegraph.

Использование cargo flamegraph​

Это позволяет визуализировать дерево вызовов и время, затраченное на каждую функцию.

# Сборка с отладочными символами, но с оптимизациями
cargo flamegraph --bin sora-core-tests -- --bench
  1. Узкое место №1: Аллокации. Ищите высокие графы в Vec::with_capacity или String::from_utf8. В Phase 2 мы минимизировали их через Zero-copy.
  2. Узкое место №2: Размаскирование. Для Radiotap-заголовков требуется много побитовых операций, которые можно ускорить через SIMD.

2. Профилирование Python-слоя (AsyncIO)​

В Python-слое критична не частота CPU, а отзывчивость Event Loop.

Использование viztracer​

viztracer идеально подходит для SORA, так как он умеет отслеживать одновременно AsyncIO корутины и нативные системные потоки Rust.

viztracer --attach_installed_ret -m sora scan -a wlan0mon profile.toml

Анализ графиков:​

  • Event Gap: Если между итерациями цикла poll_events в TUI проходит более 100 мс — значит, база данных (SQLite) или плагины блокируют основной поток.
  • GC Overhead: Всплески активности сборщика мусора Python при передаче больших объемов данных из Rust.

3. Замер IPC Overhead (Методология)​

Для точного измерения задержки маршалинга через PyO3 мы используем методику "Сквозного таймстемпинга".

Формула задержки:​

Latency_IPC = T_Python_Recv - T_Rust_Send

Как измерить:​

  1. Rust: Добавьте поле timestamp_ns в SoraEvent непосредственно перед self.high_tx.send().
  2. Python: Сохраните time.time_ns() сразу после возврата из event_receiver.poll_high().
  3. Анализ: Если задержка более 2ms, это критический сигнал о перегрузке очереди или неэффективном копировании строк MAC-адресов.

4. Performance Tuning Guide (Kernel Level)​

Для работы в режиме High-Load (мониторинг сотен сетей в реальном времени) стандартные буферы ядра Linux могут стать узким местом. Рекомендуется применить следующие настройки sysctl.

Оптимизация сетевого стека​

Добавьте в /etc/sysctl.conf:

# Увеличение максимального размера буфера приема для RAW-сокетов
net.core.rmem_max = 33554432
net.core.rmem_default = 33554432

# Увеличение очереди пакетов для интерфейса
net.core.netdev_max_backlog = 10000
  • rmem_max: Позволяет ядру удерживать больше пакетов в очереди AF_PACKET, если Rust-ядро кратковременно занято (например, при записи тяжелого блока в PCAP).
  • netdev_max_backlog: Предотвращает отбрасывание пакетов на уровне драйвера при резких всплесках трафика.
